# Content Production Plan

## Purpose

This document translates the product, journey, course, and assignment framework into a practical production plan for creating real content for the MVP.

It answers five questions:

1. What content should be produced first
2. In what order it should be produced
3. What a “ready” lesson looks like
4. How to avoid scope drift during production
5. How to organize content work so the platform fills up systematically rather than chaotically

---

## Production Principle

The MVP should not aim to create “a lot of content”.
It should aim to create **a small, coherent, role-aware, reusable learning core**.

The correct goal is:

- one strong common base for all users;
- one minimal but useful role path for each MVP role;
- consistent task and feedback quality across the whole platform.

This means:

- fewer modules, but tighter quality;
- fewer lessons, but clearer purpose;
- fewer tasks, but better alignment with real work.

---

## Main Production Priorities

### Priority 0 — Common base for all users

This is the most important content block in the whole product.

Reason:
all later role tracks depend on it.

Without a strong common base, role tracks become inconsistent, because users will enter them with very different levels of prompting, evaluation, and AI usage habits.

The common base should include:

1. What AI is useful for in work
2. How to formulate a task for a model
3. How to improve a prompt iteratively
4. How to check model output critically
5. How to work safely and not trust AI blindly
6. How to research tools and continue learning independently

---

### Priority 1 — One minimal role track for each MVP role

After the common base is stable, produce one compact track for each role:

- Developer
- Analyst
- Marketer
- Designer
- Tester

The goal is not to cover the whole profession.
The goal is to show:

- common work situations in this role;
- where AI helps;
- how to use AI well in those situations;
- where the limits are.

Each role track should feel like:

> “Now I understand how AI can actually help me in my specialty.”

---

### Priority 2 — Supporting materials

After base and role tracks exist, produce supporting content:

- optional methodical reading;
- examples of good prompts;
- examples of weak prompts and why they fail;
- tool discovery notes;
- extension tasks;
- mini case libraries.

These materials should support the core learning path, not replace it.

---

## Recommended Production Order

### Phase 1 — Build the common base

Recommended order:

1. Intro to AI-assisted work
2. What makes a good prompt
3. How to refine a prompt
4. How to evaluate AI output
5. Safety, limits, and verification
6. Tool research and independent learning

This phase must be completed first.

---

### Phase 2 — Build the first role version for each MVP role

Recommended order of production:

1. Developer
2. Analyst
3. Marketer
4. Designer
5. Tester

This order is recommended because developer and analyst scenarios are usually easier to formalize and evaluate consistently, while designer and tester flows may need more nuance in feedback and examples.

If team constraints are different, the order may change, but the rule remains:

- complete the whole role track before starting too many parallel unfinished tracks.

---

### Phase 3 — Improve depth and polish

After all tracks exist in MVP form:

- strengthen weak lessons;
- improve AI evaluation prompts;
- add optional reading;
- add richer examples;
- improve wording and flow;
- improve internal consistency between lessons.

---

## Suggested MVP Content Volume

### Common base

Recommended MVP size:

- 1 course
- 5–6 lessons
- each lesson contains:
  - 1 short explanation block;
  - 1 small knowledge check or test;
  - 1 practical task with AI feedback

This gives a strong, compact onboarding core.

---

### Role tracks

Recommended MVP size per role:

- 1 course per role
- 3–4 lessons per course
- each lesson contains:
  - 1 contextual explanation block;
  - 1 practice-oriented task;
  - optionally a mini test if needed

This is enough to create meaningful specialization without exploding production scope.

---

## Recommended Content Unit Structure

Each lesson should follow the same production logic.

### Lesson structure

1. Why this matters in the role
2. Short concept explanation
3. Example of a good approach
4. Example of a weak approach or common mistake
5. Practice task
6. AI feedback
7. Short takeaway

This creates consistency for users and makes admin-side production easier.

---

## Production Template for a Lesson

Every new lesson should be written using the following template.

### Lesson metadata

- Course name
- Role
- Module
- Lesson title
- Lesson goal
- Skill trained
- Lesson type
- Required / optional
- Unlock condition

### Lesson body

- Introduction: why this matters
- Core explanation
- Example 1
- Example 2 or anti-example
- Practice task
- Success criteria
- Typical mistakes
- Suggested AI feedback rules
- Completion rule

---

## Production Template for a Task

Every task should include:

- task title;
- role context;
- user objective;
- expected answer format;
- what the AI checker should evaluate;
- what should count as weak, acceptable, and strong;
- what mistakes are common;
- whether retries are allowed and how feedback should guide revision.

## Role-by-Role Production Focus

### 1. Developer

Start with practical, high-value scenarios:

- understanding unfamiliar code;
- writing or improving technical prompts;
- documenting code or systems;
- debugging with AI support;
- breaking down coding tasks into steps.

Avoid:

- full computer science education;
- broad programming theory;
- abstract algorithm courses unrelated to AI usage.

---

### 2. Analyst

Focus on AI support in analytical workflow:

- structuring questions;
- clarifying analytical goals;
- drafting SQL or query logic;
- summarizing findings;
- improving interpretation and communication.

Avoid:

- full statistics training from scratch;
- full BI education;
- topics not tied to AI-assisted analytical work.

---

### 3. Marketer

Focus on AI as a content and research assistant:

- generating variants of messages;
- audience-aware prompt design;
- campaign brainstorming;
- research support;
- faster drafting and iteration.

Avoid:

- full marketing education from zero;
- long theory blocks about marketing not tied to AI use.

---

### 4. Designer

Focus on AI as an ideation and reference assistant:

- reference generation;
- concept variation;
- style articulation;
- moodboard direction;
- task decomposition for creative work.

Avoid:

- full design education;
- deep art theory unrelated to AI-assisted practice.

---

### 5. Tester

Focus on AI in quality and validation workflows:

- writing test cases;
- generating edge cases;
- clarifying feature behavior;
- checking negative scenarios;
- structuring bug reports and QA notes.

Avoid:

- full QA education from zero;
- broad software testing theory not tied to AI usage.

---

## Content Quality Rules

Every piece of content must pass these checks.

### Check 1 — Relevance

Does this teach AI usage in the user’s specialty?

If not, it should not be in the main track.

---

### Check 2 — Practicality

Can the learner apply this to a real work-like task?

If not, it is likely too abstract.

---

### Check 3 — Role fit

Is this for everyone or for a specific role?

- if for everyone → common base
- if role-specific → role track
- if optional / deep reading → methodical materials

---

### Check 4 — Scope discipline

Does this accidentally turn into full education in the profession itself?

If yes, narrow it back to AI-assisted application.

---

### Check 5 — Feedback readiness

Can the task be meaningfully checked by the AI feedback system?

If not, rewrite the task or simplify it.

---

## Recommended Production Workflow

For each lesson:

1. Define role and lesson goal
2. Identify the real work scenario
3. Define the skill trained
4. Write a short explanation block
5. Add one strong example
6. Add one weak example
7. Write the task
8. Write success criteria
9. Define common mistakes
10. Define AI feedback behavior
11. Review for scope drift
12. Publish draft to admin builder
13. Test from the learner perspective
14. Revise

This workflow should be repeated consistently.

## Production Backlog for MVP

### Block A — Must produce first

1. Common base course
2. Developer track
3. Analyst track
4. Marketer track
5. Designer track
6. Tester track

---

### Block B — Produce after MVP core exists

1. optional methodical materials
2. examples library
3. tool discovery lists
4. advanced challenge tasks
5. cross-role mini cases

---

### Block C — Do not produce yet

1. large academic theory sections
2. profession-from-zero content
3. too many deep optional branches
4. over-specialized subtracks
5. content with unclear AI relevance

---

## Review Cycle

Each lesson should be reviewed at least once across four dimensions:

1. Product fit
2. Role fit
3. AI feedback quality
4. UX clarity

Questions to ask during review:

- Is the lesson useful to the intended user?
- Is it too broad or too theoretical?
- Is the task understandable?
- Can the feedback system evaluate it fairly?
- Does the lesson clearly move the user forward?

---

## Definition of Done for a Lesson

A lesson is ready when:

- it has a clear role and goal;
- it fits the project scope;
- it teaches AI usage, not profession-from-zero content;
- it includes explanation, example, task, and evaluation logic;
- it has clear success criteria;
- it has clear common mistakes;
- it can be checked by the AI system;
- it fits the course flow and unlock logic.

---

## Definition of Done for a Role Track

A role track is MVP-ready when:

- it includes 3–4 coherent lessons;
- each lesson trains a real role-relevant AI usage skill;
- tasks feel like realistic work situations;
- content stays inside project scope;
- progression feels sequential and not random;
- tasks can be checked consistently;
- the user finishes the track with clear practical benefit.

---

## Final Production Rule

Do not optimize for volume.

Optimize for:

- coherence;
- role relevance;
- practical value;
- consistency of feedback;
- clarity of progression.

The strongest MVP is not the platform with the most content.
It is the platform where every lesson clearly belongs, clearly helps, and clearly moves the learner forward.
