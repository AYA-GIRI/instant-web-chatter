# Task Library — Internium / Центр разработки стажировок и инноваций с GPT

## 1. Назначение документа

Этот документ переводит `content-matrix.md` и `assignment-spec.md` в уже готовый пул заданий для практикума.

Его задача — дать команде не абстрактную идею «нужно сделать задания», а стартовую библиотеку из типовых упражнений, которые можно:

- сразу перенести в конструктор практикума;
- использовать как основу для MVP;
- адаптировать под разные роли без придумывания с нуля;
- применять как шаблон для новых уроков и модулей.

Документ не пытается покрыть весь будущий контент проекта. Он фиксирует **первую рабочую библиотеку заданий**, на которой уже можно собирать цельный практикум.

---

## 2. Как использовать библиотеку

Каждое задание ниже можно использовать одним из трёх способов:

### 2.1. Как готовое задание

Берёшь почти без изменений и переносишь в практикум.

### 2.2. Как шаблон

Сохраняешь структуру, но меняешь рабочий контекст, роль или пример.

### 2.3. Как основу для серии

Из одного задания делаешь серию из 2–3 уровней:

- базовый;
- средний;
- продвинутый.

---

## 3. Принципы отбора заданий

В библиотеку включаются только такие задания, которые:

- учат применению AI в работе;
- строятся вокруг реальной рабочей задачи;
- тренируют осмысленное использование AI, а не магическое мышление;
- подходят под текущий scope проекта;
- можно проверить через AI-фидбек и понятные критерии.

Не включаются задания, которые:

- обучают профессии с нуля;
- проверяют академическую теорию ради теории;
- слабо связаны с ролью пользователя;
- требуют сложной ручной проверки, если её нельзя заменить разумным AI-фидбеком.

---

## 4. Рекомендуемая структура MVP-практикума

Для первого нормального наполнения платформы рекомендуется такая структура:

### Общая база
- 4 модуля
- по 2–3 задания в каждом модуле
- всего 10–12 заданий

### Ролевые треки
- 5 ролей
- по 3 модуля на роль для первого этапа
- по 2 задания на модуль
- всего 6 заданий на роль

### Итого для первого сильного MVP
- общая база: 10–12 заданий
- роли: 30 заданий
- всего: **40–42 задания**

Это уже достаточно, чтобы проект ощущался как собранная обучающая система, а не как демо из пары страниц.

# 5. Общая база — библиотека заданий

## BASE1. Понимание рабочей задачи для AI

### BASE1_TASK_01 — Превратить размытый запрос в рабочий

**Тип:** T2 — улучшить плохой промпт  
**Цель:** научить видеть, почему расплывчатый запрос даёт слабый результат.

**Контекст:** пользователь пишет модели:  
> «Помоги мне с задачей по работе»

**Что нужно сделать:**
Переписать запрос так, чтобы модель поняла:
- кто пользователь;
- какая у него роль;
- какую задачу он решает;
- какой результат ему нужен.

**Критерии хорошего ответа:**
- есть роль;
- есть рабочая ситуация;
- есть цель;
- есть ожидаемый формат ответа.

---

### BASE1_TASK_02 — Отделить задачу от шума

**Тип:** T6 — разложить задачу на шаги с AI  
**Цель:** научить не бросать в модель весь хаос сразу.

**Контекст:** пользователь хочет с помощью AI «подготовиться к созвону, написать черновик письма и придумать, как оформить результат». 

**Что нужно сделать:**
Разбить задачу на 3 отдельных AI-запроса и объяснить, почему так лучше.

**Критерии хорошего ответа:**
- задачи разделены;
- у каждого шага понятная цель;
- есть логика порядка;
- видно, что AI используется как ассистент по этапам.

# 7. Как собирать задания в курс

Для первого этапа можно использовать такой принцип сборки:

## Формула модуля
- 1 короткий инфоблок
- 1 мини-тест
- 2 практических задания
- 1 вывод / правило

## Формула трека
- 1 вводный модуль по роли
- 2 прикладных модуля
- 1 модуль про поиск инструментов и самостоятельный growth

---

# 8. Что стоит делать сразу после этой библиотеки

После этого документа самый логичный следующий шаг — собрать:

## 8.1. `course-outlines.md`

Там уже фиксируется:
- какие именно курсы входят в MVP;
- как называются модули;
- в каком порядке идут уроки;
- сколько в каждом модуле блоков.

## 8.2. `ai-feedback-rubrics.md`

Там фиксируется:
- по каким критериям модель оценивает ответы;
- какие подсказки давать;
- какие ошибки считать критичными;
- как отличать «почти готово» от «надо переделать».
