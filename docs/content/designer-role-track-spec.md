# Designer Role-Track Spec

## Purpose

This document defines the **Designer track**.

Its purpose is to answer:

- what this course should teach;
- what lessons it should contain;
- what practical tasks are needed;
- which task types are already supported by the current builder;
- what should be reviewed after the course is created and tested.

This document should be used **before** creating the Designer course in the practicum builder.

---

## Course Identity

### Recommended course title

**AI for Designers: Practical Design Workflows**

Alternative titles:
- AI for Designers: First Practical Workflows
- Practical AI for Junior Designers
- AI Assistance in Design Work

---

## Course Goal

After finishing this course, the learner should be able to:

- use AI to clarify a design task before execution;
- use AI to generate and refine visual directions, references, and concept variants;
- use AI to articulate style and design constraints more clearly;
- critically evaluate AI-generated visual or textual design suggestions;
- use AI to support moodboards, exploration, and explanation without outsourcing design judgment;
- use AI safely without losing brand fit, product logic, or design intent.

This course is **not** meant to teach design from zero.
It is meant to teach **how a designer can apply AI in realistic work-like scenarios**.

---

## Course Position In Product Logic

This course belongs to:

- `course_category = role_track`

It should become available **after** the common base is completed.

It is intended for:

- junior designers;
- interns with basic design knowledge;
- learners who already passed the common base and understand:
  - what AI is,
  - how to write prompts,
  - how to iterate,
  - why verification matters.

---

## Core Course Principle

This role track should not become:
- a full design curriculum;
- a visual theory-from-zero course;
- a “generate pretty things” course;
- a prompt-writing-only course.

Instead, it should teach:
- realistic AI-assisted design workflows;
- stronger articulation of design intent;
- useful exploration without blind trust;
- critical thinking about generated output.

# Lesson 1 — Turn a vague design request into a clear task

## Goal

Teach the learner to clarify a design request before asking AI for ideas or visual directions.

## What the learner should understand

- vague design tasks produce vague output;
- useful requests need product context, audience, medium, and constraints;
- AI helps best when the designer has already framed the task well.

## Practical task type

**Improve a vague request**

## Suggested task

You are given a vague request:

> “Make a cool design for our product.”

Rewrite it into a stronger design request for AI so that it becomes clear:
- what product or screen is involved,
- who the audience is,
- what visual goal matters,
- what constraints exist,
- what output is expected.

## Success criteria

- clarifies context;
- clarifies audience or usage;
- clarifies design goal;
- includes constraints;
- avoids generic “make it nice”.

## Common mistakes

- still vague;
- no audience or medium;
- no constraints;
- no expected output.

## Builder fit

**Fully supported by current builder.**

# Lesson 2 — Use AI to generate design directions and references

## Goal

Teach the learner to use AI for structured visual exploration instead of random inspiration.

## What the learner should understand

- AI can help generate style directions and reference categories;
- exploration should still be anchored in product and user needs;
- designers should guide and filter results.

## Practical task type

**Write a structured request**

## Suggested task

Write a request to AI that will help generate 3 visual directions or reference groups for a design task.

The request should specify:
1. what is being designed;
2. who it is for;
3. what style or mood should be explored;
4. what should differ across the directions;
5. how the result should be structured.

## Success criteria

- design context is clear;
- audience is clear;
- style or mood is specified;
- asks for multiple distinct directions;
- output structure is requested.

## Common mistakes

- no audience;
- no design context;
- all directions would end up the same;
- no structure.

## Builder fit

**Fully supported by current builder.**

# Lesson 3 — Critically evaluate weak AI-generated design suggestions

## Goal

Teach the learner to identify when AI-generated design ideas are generic, inconsistent, or not grounded in real product logic.

## What the learner should understand

- generated design suggestions can be visually plausible but product-weak;
- designers must judge usability, clarity, and fit;
- AI output should be filtered, not adopted blindly.

## Practical task type

**Analyze and critique AI output**

## Suggested task

You are given an AI-generated design suggestion or concept summary.

Describe:
1. what in it looks useful;
2. what feels generic, weak, or inconsistent;
3. what should be checked against product logic or user needs;
4. why you should not adopt the suggestion blindly.

## Success criteria

- identifies useful and weak parts;
- ties critique to product logic or user needs;
- names what should be checked manually;
- avoids vague “it’s bad” style critique.

## Common mistakes

- only praise or only rejection;
- no product logic;
- no manual verification points;
- too abstract.

## Builder fit

**Supported by current builder.**

# Lesson 4 — Use AI to articulate style and constraints clearly

## Goal

Teach the learner to express design style and constraints in a way that makes AI output more useful.

## What the learner should understand

- design requests need constraints, not just aesthetic adjectives;
- audience, product tone, accessibility, and platform matter;
- stronger articulation improves AI usefulness.

## Practical task type

**Improve a weak request**

## Suggested task

You are given a weak request:

> “Make it modern and minimal.”

Rewrite it so that AI understands:
- what is being designed,
- for whom,
- what “modern and minimal” should mean in this context,
- what constraints or exclusions matter,
- what kind of output is expected.

## Success criteria

- replaces vague style language with meaningful context;
- includes audience and medium;
- includes constraints;
- makes the request more actionable.

## Common mistakes

- still relies only on aesthetic adjectives;
- no product context;
- no constraints;
- no expected output.

## Builder fit

**Fully supported by current builder.**

# Lesson 5 — Use AI to support moodboards and early concept exploration

## Goal

Teach the learner to use AI to support concept exploration without mistaking output for a final design decision.

## What the learner should understand

- AI can help expand early concept space;
- moodboards and concept sets still need human selection and interpretation;
- good exploration includes comparison and filtering.

## Practical task type

**Describe a structured workflow**

## Suggested task

Describe how you would use AI to support the early exploration stage of a design task.

Your answer should include:
1. what context you would give AI;
2. how you would ask for visual directions or references;
3. how you would compare and filter outputs;
4. how you would turn the results into a usable design direction.

## Success criteria

- includes structured exploration stages;
- includes context input;
- includes filtering/comparison logic;
- shows that designer judgment remains central.

## Common mistakes

- too abstract;
- no filtering logic;
- no comparison stage;
- treats generated outputs as final.

## Builder fit

**Supported by current builder.**

# Lesson 6 — Use AI to explain design rationale and handoff more clearly

## Goal

Teach the learner to use AI for explaining design decisions, rationale, or handoff notes in a clearer way.

## What the learner should understand

- AI can help structure rationale and handoff explanations;
- audience matters (designer, PM, developer, stakeholder);
- explanations should not invent reasons that were never part of the design process.

## Practical task type

**Write a structured request**

## Suggested task

Write a request to AI that will help draft a short explanation of a design decision or a handoff note.

The request should specify:
1. who the audience is;
2. what is being explained;
3. what sections or structure are needed;
4. that the AI should not invent reasons not grounded in the actual design.

## Success criteria

- audience is clear;
- explanation target is clear;
- structure is requested;
- instruction against invented rationale is included.

## Common mistakes

- no audience;
- generic explanation request;
- no structure;
- no instruction against making things up.

## Builder fit

**Fully supported by current builder.**

# Lesson 7 — Safe and responsible AI use in design

## Goal

Teach the learner to think about design quality, brand consistency, and responsible use of AI in design workflows.

## What the learner should understand

- AI can produce off-brand, inaccessible, or product-incoherent output;
- designers remain responsible for usability, clarity, and consistency;
- useful AI usage must stay inside real design intent.

## Practical task type

**Analyze a scenario**

## Suggested task

Describe:
1. one case where it is useful for a designer to use AI;
2. one case where using AI directly could create a risk for usability, brand consistency, or design quality.

For the risky case, explain:
- what the risk is;
- what should be checked or corrected;
- what should not be done directly.

## Success criteria

- gives realistic design situations;
- names a concrete risk;
- explains safe follow-up behavior;
- avoids blind trust language.

## Common mistakes

- too abstract;
- no concrete risk;
- no safe action;
- ignores product or brand fit.

## Builder fit

**Supported by current builder.**

# What To Do After The Designer Course Is Written

After the Designer role track is fully drafted or entered into the builder:

## 1. Run a post-course content production audit

Answer:
- which lessons were easy to create?
- which lesson types created friction?
- which task formats felt unnatural in the current builder?
- what repeated manual work appeared?
- what could be improved without changing the whole system?

## 2. Compare the Designer postmortem with earlier postmortems

Ask:
- did the same friction appear again?
- are the same task patterns awkward?
- does the builder distort this course in the same places?

## 3. Decide whether builder changes are actually needed

### Outcome A — Builder is enough
If the course was created with only minor inconvenience:
- move to the next role track;
- keep notes for later improvements.

### Outcome B — Small builder improvements are justified
If repeated friction clearly appears again:
- make a small, evidence-based builder update;
- keep changes backward-compatible.

### Outcome C — A genuinely missing task pattern exists
If an important pattern feels distorted:
- document exactly what the missing pattern is;
- explain why current model is not enough;
- then design the smallest extension.

## 4. Only then plan the next builder update

The builder should evolve like this:

- content first,
- pain observed,
- repeated pattern confirmed,
- minimal improvement,
- test with next course.
