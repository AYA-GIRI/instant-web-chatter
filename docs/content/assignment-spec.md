# Assignment Spec — Internium / Центр разработки стажировок и инноваций с GPT

## 1. Назначение документа

Этот документ переводит `content-matrix.md` в правила создания конкретных заданий, уроков и AI-проверки.

Его задача — сделать так, чтобы:

- задания не придумывались хаотично;
- тексты уроков писались по одному стандарту;
- AI-проверка проверяла не «что угодно», а конкретные критерии;
- задания для разных ролей были разными по смыслу, но одинаковыми по качеству;
- проект не расползался в обучение профессиям с нуля.

Этот документ нужен как мост между:

- продуктовой рамкой проекта;
- матрицей контента по ролям;
- реальным наполнением практикума.

---

## 2. Как использовать этот документ

Каждый новый урок и каждое новое задание должны проходить через такую цепочку:

1. Определить роль.
2. Выбрать рабочий сценарий.
3. Понять, какой AI-навык тренируется.
4. Выбрать формат задания.
5. Прописать критерии хорошего ответа.
6. Прописать типовые ошибки.
7. Настроить AI-фидбек по этим критериям.

Если какой-то блок не проходит эту цепочку, его не стоит добавлять в практикум.

---

## 3. Главный принцип заданий

Платформа обучает **не профессии с нуля**, а **применению AI в конкретной профессиональной задаче**.

Поэтому хорошее задание должно быть не таким:

> «Расскажи, что такое маркетинг»

и не таким:

> «Объясни, как работают attention-слои в LLM»

а таким:

> «Составь запрос к AI, который поможет маркетологу быстро подготовить 3 варианта оффера под разную аудиторию»

---

## 4. Формула хорошего задания

Каждое задание строится по формуле:

**роль -> рабочая ситуация -> польза AI -> тренируемый навык -> формат ответа -> критерии качества**

### Пример

- **Роль:** разработчик
- **Ситуация:** нужно быстро понять чужой модуль
- **Польза AI:** получить объяснение логики и выделить слабые места
- **Навык:** правильно формулировать запрос на анализ кода
- **Формат ответа:** текстовый промпт
- **Критерии качества:** есть контекст, цель, формат результата, ограничения

---

## 5. Типы блоков в уроке

В текущем проекте уже есть логика блочной сборки уроков. Для контентной стандартизации рекомендуются такие типы блоков:

### 5.1. Инфоблок / Markdown

Используется для:

- объяснения, зачем тема нужна;
- краткой теории;
- примеров хорошего и плохого подхода;
- фиксации типичных ошибок.

### 5.2. Мини-тест

Используется для:

- проверки, понял ли пользователь базовую идею;
- отсечения грубых ошибок до практики;
- закрепления терминов и логики.

### 5.3. Практическое AI-задание

Главный блок практикума.

Используется для:

- написания промпта;
- улучшения ответа;
- анализа чужого AI-ответа;
- выбора инструмента;
- сравнения подходов.

### 5.4. Рефлексия / вывод

Используется для:

- закрепления результата;
- формулировки правила, которое пользователь уносит с собой;
- перехода к следующему шагу.

---

## 6. Стандартная структура урока

Каждый урок в практикуме должен по возможности повторять одну и ту же структуру.

### Шаблон урока

1. **Зачем это нужно в работе**
2. **Краткое объяснение**
3. **Пример хорошего и плохого подхода**
4. **Мини-тест или проверка понимания**
5. **Практическое задание**
6. **AI-фидбек**
7. **Вывод / правило урока**

### Что это даёт

- пользователю легче ориентироваться;
- автору контента проще собирать уроки;
- админке легче стандартизировать наполнение;
- AI-проверка становится предсказуемой.

---

## 7. Стандартная структура задания

Каждое практическое задание должно иметь одинаковый каркас.

## Шаблон задания

### 1. ID задания
Уникальный идентификатор.

Пример: `DEV2_TASK_01`

### 2. Роль
Для кого предназначено задание.

Пример: `developer`

### 3. Модуль
К какому модулю относится.

Пример: `DEV2. Генерация черновиков и доработка кода`

### 4. Название
Короткое рабочее название.

Пример: `Промпт на генерацию черновика функции`

### 5. Цель задания
Что должен уметь пользователь после выполнения.

### 6. Рабочий контекст
Реалистичная мини-ситуация из работы.

### 7. Что должен сделать пользователь
Конкретное действие.

### 8. Формат ответа
Что именно вводит пользователь:

- промпт;
- улучшенный промпт;
- анализ AI-ответа;
- сравнение инструментов;
- список шагов;
- структурированный текст.

### 9. Критерии хорошего ответа
По каким признакам AI и админ понимают, что ответ качественный.

### 10. Типовые ошибки
Какие слабые места чаще всего будут встречаться.

### 11. Подсказка (опционально)
Мягкий намёк без раскрытия полного решения.

### 12. Эталонный черновик / reference answer
Не обязательно показывать пользователю, но полезно для настройки проверки.

---

## 8. Разрешённые форматы заданий

Для MVP достаточно ограничить практикум несколькими типами заданий.

### T1. Составить промпт

Пользователь пишет запрос под рабочую задачу.

**Подходит для:**
- базы;
- всех ролей.

### T2. Улучшить плохой промпт

Пользователь получает слабый запрос и делает его лучше.

**Подходит для:**
- общей базы;
- закрепления структуры хорошего запроса.

### T3. Улучшить AI-ответ

Пользователь должен понять, как уточнить модель и добиться лучшего результата.

### T4. Найти ошибки в AI-ответе

Пользователь анализирует недостоверный, поверхностный или плохо применимый ответ.

### T5. Выбрать AI-инструмент под задачу

Пользователь сравнивает типы инструментов и обосновывает выбор.

### T6. Разложить рабочую задачу на шаги с AI

Пользователь показывает, как AI встраивается в рабочий процесс, а не просто даёт один ответ.

---

## 9. Базовые критерии хорошего ответа

Независимо от роли, многие задания можно оценивать по общему набору критериев.

### 9.1. Для промптов

Хороший промпт обычно содержит:

- понятную цель;
- контекст задачи;
- ограничения;
- ожидаемый формат ответа;
- достаточную конкретику;
- связь с рабочей ситуацией.

### 9.2. Для анализа AI-ответов

Хороший анализ должен:

- замечать фактические или логические слабости;
- отличать полезный черновик от готового результата;
- указывать, что нужно перепроверить;
- объяснять, почему ответ не полностью подходит.

### 9.3. Для выбора инструментов

Хороший ответ должен:

- исходить из рабочей задачи, а не из популярности инструмента;
- учитывать тип результата;
- учитывать ограничения роли;
- объяснять, почему выбран именно этот класс инструментов.

---

## 10. Типовые ошибки пользователей

Эти ошибки нужно учитывать и в текстах уроков, и в AI-фидбеке.

### Общие ошибки

- слишком общий запрос без контекста;
- отсутствие цели;
- отсутствие формата ответа;
- попытка получить идеальный результат за один запрос;
- слепое доверие ответу модели;
- попытка делегировать AI задачу без понимания, что проверять;
- подмена прикладного задания теорией.

### Ошибки по ролям

#### Разработчик
- просит «написать код», не описав окружение и ограничения;
- не указывает язык, фреймворк, входы/выходы;
- не проверяет безопасность и корректность решения.

#### Аналитик
- не формулирует метрику успеха;
- не указывает формат вывода;
- не различает summary и аналитический вывод.

#### Маркетолог
- не указывает аудиторию;
- не задаёт tone of voice;
- просит «сделать текст», не задав канал и цель.

#### Дизайнер
- не описывает стиль;
- не задаёт ограничения по формату;
- не различает референсы, концепт и финальный дизайн.

#### Тестировщик
- не указывает систему, сценарий и критерий бага;
- не формулирует ожидаемое поведение;
- просит «протестировать всё» без рамки.

---

## 11. Правила AI-проверки

AI-проверка должна быть не «оценкой личности», а проверкой структуры решения.

### AI-фидбек должен:

- отмечать, что уже хорошо;
- указывать 1–3 главных недочёта;
- давать конкретный вектор улучшения;
- не выдавать сразу идеальный ответ вместо пользователя;
- поддерживать итеративную доработку.

### AI-фидбек не должен:

- просто говорить «неправильно»;
- быть слишком общим;
- заменять собой обучение;
- уходить в длинную лекцию;
- оценивать ответ по скрытым критериям, которые не объяснены в уроке.

### Рекомендуемый формат фидбека

1. Что получилось хорошо
2. Что улучшить
3. Чего не хватает
4. Что попробовать в следующей версии

---

## 12. Что включать и что не включать в задания

## Включать

- реальные рабочие мини-ситуации;
- прикладные сценарии по роли;
- короткий контекст, понятный новичку;
- действия, где AI действительно даёт ускорение;
- формирование привычки проверять и уточнять;
- поиск и сравнение инструментов.

## Не включать

- длинные академические вопросы;
- обучение профессии с нуля;
- задачи, не связанные с использованием AI;
- задания, где пользователь просто пересказывает теорию;
- слишком большие кейсы, которые трудно пройти в одном экране;
- оценку по вкусу вместо явных критериев.

## 13. Мини-шаблоны заданий по ролям

## 13.1. Разработчик

### Шаблон

**Контекст:** Ты джун-разработчик. Тебе дали фрагмент кода, и нужно быстро понять его назначение и потенциальные слабые места.

**Что сделать:** Напиши промпт для AI, который поможет:
1. кратко объяснить логику кода;
2. выделить возможные проблемы;
3. предложить структуру документации.

**Что проверяем:**
- есть ли контекст;
- задана ли цель;
- указан ли формат ответа;
- учтена ли практическая польза результата.

## 13.2. Аналитик

**Контекст:** Нужно быстро разобраться в таблице метрик и подготовить черновик выводов.

**Что сделать:** Напиши промпт для AI, который поможет структурировать наблюдения и отделить факты от интерпретаций.

## 13.3. Маркетолог

**Контекст:** Нужно подготовить несколько вариантов сообщения под разные сегменты аудитории.

**Что сделать:** Составь промпт, который поможет получить 3 разные версии текста под разные сегменты и каналы.

## 13.4. Дизайнер

**Контекст:** Нужно быстро собрать направления референсов для интерфейса или визуальной концепции.

**Что сделать:** Составь промпт, который поможет получить несколько визуальных направлений с разным стилевым акцентом.

## 13.5. Тестировщик

**Контекст:** Нужно быстро подготовить тест-кейсы для типового пользовательского сценария.

**Что сделать:** Составь промпт для AI, который поможет выделить позитивные, негативные и пограничные сценарии.

---

## 14. Definition of Done для нового задания

Новое задание считается готовым, если:

- оно привязано к конкретной роли или базовому модулю;
- в нём есть реалистичный рабочий контекст;
- понятен тренируемый навык;
- определён формат ответа;
- прописаны критерии хорошего ответа;
- прописаны типовые ошибки;
- AI-фидбек можно настроить не абстрактно, а по критериям;
- задание не уводит проект в обучение профессии с нуля.
