# Marketer Role-Track Spec

## Purpose

This document defines the **Marketer track**.

Its purpose is to answer:

- what this course should teach;
- what lessons it should contain;
- what practical tasks are needed;
- which task types are already supported by the current builder;
- what should be reviewed after the course is created and tested.

This document should be used **before** creating the Marketer course in the practicum builder.

---

## Course Identity

### Recommended course title

**AI for Marketers: Practical Marketing Workflows**

Alternative titles:
- AI for Marketers: First Practical Workflows
- Practical AI for Junior Marketers
- AI Assistance in Marketing Work

---

## Course Goal

After finishing this course, the learner should be able to:

- use AI to clarify marketing tasks before execution;
- use AI to generate and refine audience-aware messaging;
- use AI to structure campaign and content ideas;
- critically evaluate AI-generated marketing texts and claims;
- use AI to accelerate research, copy drafts, and idea generation without trusting output blindly;
- use AI safely without producing misleading, generic, or brand-inconsistent content.

This course is **not** meant to teach marketing from zero.
It is meant to teach **how a marketer can apply AI in realistic work-like scenarios**.

---

## Course Position In Product Logic

This course belongs to:

- `course_category = role_track`

It should become available **after** the common base is completed.

It is intended for:

- junior marketers;
- interns with basic marketing knowledge;
- learners who already passed the common base and understand:
  - what AI is,
  - how to write prompts,
  - how to iterate,
  - why verification matters.

---

## Core Course Principle

This role track should not become:
- a full marketing curriculum;
- a copywriting-from-zero course;
- a branding theory course;
- a prompt-writing-only course.

Instead, it should teach:
- realistic AI-assisted marketing workflows;
- audience-aware thinking;
- structured request design;
- critical evaluation of AI-generated content.

# Lesson 1 — Turn a vague marketing task into a clear request

## Goal

Teach the learner to transform an unclear marketing task into a request AI can actually help with.

## What the learner should understand

- vague requests lead to generic output;
- marketing requests need audience, channel, purpose, tone, and format;
- AI helps best when the task is framed clearly.

## Practical task type

**Improve a vague request**

## Suggested task

You are given a vague task:

> “Make a marketing text for our product.”

Rewrite it into a stronger request for AI so that it becomes clear:
- who the audience is,
- what channel the text is for,
- what the goal is,
- what tone is needed,
- what output format is expected.

## Success criteria

- clarifies audience;
- clarifies channel or placement;
- clarifies business goal;
- includes tone or style guidance;
- avoids generic phrasing.

## Common mistakes

- still vague;
- no audience;
- no channel;
- no purpose beyond “write text”.

## Builder fit

**Fully supported by current builder.**

# Lesson 2 — Generate audience-aware messaging

## Goal

Teach the learner to ask AI for text variants that fit a real audience instead of producing bland generic copy.

## What the learner should understand

- one message does not fit every audience;
- audience context changes wording, hooks, and examples;
- AI can generate variants, but the marketer must choose and refine.

## Practical task type

**Write a structured request**

## Suggested task

Write a request to AI that will help generate 5 message variants for a campaign.

The request should specify:
1. audience;
2. product or offer;
3. communication goal;
4. tone of voice;
5. output format.

## Success criteria

- audience is clear;
- offer or product is clear;
- communication goal is clear;
- tone is specified;
- answer format is requested.

## Common mistakes

- no audience segmentation;
- no goal;
- no tone;
- too generic to produce useful variants.

## Builder fit

**Fully supported by current builder.**

# Lesson 3 — Improve weak AI-generated copy

## Goal

Teach the learner to diagnose why AI-generated marketing copy feels weak and how to improve the request or the output.

## What the learner should understand

- AI copy can be generic, repetitive, or off-brand;
- weak output should be improved through better direction;
- marketers must identify what exactly feels wrong.

## Practical task type

**Analyze and improve weak output**

## Suggested task

You are given a weak AI-generated marketing message.

Describe:
1. what feels generic or weak;
2. what is missing for the target audience;
3. what should be improved in the next request or revision;
4. how to make the text more useful and less generic.

## Success criteria

- identifies concrete weaknesses;
- ties critique to audience or goal;
- suggests practical improvements;
- avoids vague “make it better” feedback.

## Common mistakes

- only says “too generic” without specifics;
- no audience logic;
- no concrete improvement direction;
- no relation to business goal.

## Builder fit

**Supported by current builder.**
This is not only prompt-writing.

# Lesson 4 — Use AI to structure campaign and content ideas

## Goal

Teach the learner to use AI for idea generation and structured planning rather than random brainstorming only.

## What the learner should understand

- AI can help expand idea space quickly;
- ideas should still be structured by objective, audience, and channel;
- marketers should evaluate usefulness, not just novelty.

## Practical task type

**Describe a structured workflow**

## Suggested task

Describe how you would use AI to plan initial campaign or content ideas for a new launch.

Your answer should include:
1. what input context you would give AI;
2. how you would ask for idea categories or content directions;
3. how you would select promising ideas;
4. how you would refine them for real use.

## Success criteria

- includes structured process;
- includes input context;
- includes selection/refinement logic;
- avoids “just ask AI for ideas”.

## Common mistakes

- too abstract;
- no refinement step;
- no selection logic;
- no role of audience or goal.

## Builder fit

**Supported by current builder.**

# Lesson 5 — Use AI to support marketing research and comparisons

## Goal

Teach the learner to use AI as a helper in marketing research without mistaking generated guesses for verified facts.

## What the learner should understand

- AI can help organize questions and summarize patterns;
- research still requires source checking and validation;
- AI-generated comparisons can sound plausible while being unsupported.

## Practical task type

**Describe a verification-aware request or workflow**

## Suggested task

Write a request to AI that helps you structure initial research about competitors or similar products.

The request should specify:
1. what you want to compare;
2. what dimensions matter;
3. what should be treated as hypothesis vs verified fact;
4. how the answer should be structured.

## Success criteria

- comparison target is clear;
- research dimensions are clear;
- hypothesis vs fact is acknowledged;
- asks for useful structure.

## Common mistakes

- treats AI as final source of truth;
- no comparison dimensions;
- no verification awareness;
- too broad.

## Builder fit

**Fully supported by current builder.**

# Lesson 6 — Draft summaries and reporting for stakeholders

## Goal

Teach the learner to use AI to structure stakeholder-facing summaries without overstating findings or inventing certainty.

## What the learner should understand

- stakeholder summaries need clarity and brevity;
- the audience matters;
- uncertainty and caveats should not disappear;
- AI can help with structure, not replace judgment.

## Practical task type

**Write a structured request**

## Suggested task

Write a request to AI that will help draft a short update for a stakeholder about campaign results or content performance.

The request should specify:
1. who the audience is;
2. what findings should be summarized;
3. what structure is needed;
4. what caution or uncertainty should remain visible.

## Success criteria

- audience is clear;
- findings are scoped;
- structure is requested;
- caution is preserved.

## Common mistakes

- no audience;
- too generic;
- no uncertainty;
- turns results into overconfident claims.

## Builder fit

**Fully supported by current builder.**

# Lesson 7 — Safe and responsible AI use in marketing

## Goal

Teach the learner to think about misleading claims, brand inconsistency, and careless use of AI in marketing work.

## What the learner should understand

- AI can create misleading promises, invented facts, or off-brand messaging;
- marketers remain responsible for claims and positioning;
- safety includes brand tone, factual correctness, and ethical messaging.

## Practical task type

**Analyze a scenario**

## Suggested task

Describe:
1. one case where it is useful for a marketer to use AI;
2. one case where using AI directly could create a risk for brand consistency, misleading claims, or poor communication quality.

For the risky case, explain:
- what the risk is;
- what should be checked or corrected;
- what should not be done directly.

## Success criteria

- gives realistic marketer situations;
- names a concrete risk;
- explains a safe behavior or manual check;
- avoids blind trust language.

## Common mistakes

- too abstract;
- no clear risk;
- no safe action;
- ignores brand or claim quality.

## Builder fit

**Supported by current builder.**

# What To Do After The Marketer Course Is Written

After the Marketer role track is fully drafted or entered into the builder:

## 1. Run a post-course content production audit

Answer:
- which lessons were easy to create?
- which lesson types created friction?
- which task formats felt unnatural in the current builder?
- what repeated manual work appeared?
- what could be improved without changing the whole system?

## 2. Compare the Marketer postmortem with earlier postmortems

Ask:
- did the same friction appear again?
- are the same task patterns awkward?
- does the builder distort this course in the same places?

## 3. Decide whether builder changes are actually needed

### Outcome A — Builder is enough
If the course was created with only minor inconvenience:
- move to the next role track;
- keep notes for later improvements.

### Outcome B — Small builder improvements are justified
If repeated friction clearly appears again:
- make a small, evidence-based builder update;
- keep changes backward-compatible.

### Outcome C — A genuinely missing task pattern exists
If an important pattern feels distorted:
- document exactly what the missing pattern is;
- explain why current model is not enough;
- then design the smallest extension.

## 4. Only then plan the next builder update

The builder should evolve like this:

- content first,
- pain observed,
- repeated pattern confirmed,
- minimal improvement,
- test with next course.
